# 先验证再构建

> 来源：提炼自 `~/.claude/notes/` Council Lessons 中的这些原则：GStack adoption scope、Premature integration、Verify before building、Measure before building、No test scaffolding in CI、No blanket test stubs。

## 规则

在你拥有**经验证据**之前，不要构建、移植或集成功能。关于缺口、覆盖率或用户需求的判断，必须先由真实数据验证，之后才值得投入工程工作。

## 检查清单（在开始任何构建/移植/集成前）

- [ ] **问题是否已经验证？** 你测量过真实数据上的差距，而不是凭估计吗？
- [ ] **真的有人需要吗？** 这是已有用户需求，还是你在预判一个可能永远不会出现的需求？
- [ ] **现有方案真的不够吗？** 在真实输入上比较当前方案与拟议方案，再判断是否有缺口。
- [ ] **值得长期维护吗？** 第 N+1 个功能的边际价值会迅速接近零，但维护成本会一直存在。

## 反模式

| 陷阱 | 示例 | 修正方式 |
|------|------|----------|
| 优化未测量的指标 | “覆盖率大概有 85%” 但没有审计 | 先测量，再优化 |
| 推测式搭建集成桥接 | “这两个工具应该能联动” | 等到真实用户提出需求 |
| 通过竞品差距直接移植功能 | “竞品有 X，我们没有” | 先问：没有 X，真的有人被阻塞吗？ |
| 不做对比就替代付费工具 | “我们可以自己做一个 X” | 用同一批输入同时跑两套方案，再比较结果 |
| 给非应用型仓库预先搭测试脚手架 | “这个仓库没有测试配置” | 先确认是否存在可测试代码；配置型仓库优先考虑 lint |
| 给每个模块批量生成测试桩 | “每个模块都得有测试文件” | 一个高质量模板加按需 `/tdd`，胜过 50 个 TODO 空壳 |

## 怀疑者测试

在决定开始构建前，先问自己：*"如果一个完全不了解上下文的怀疑者质疑这项工作，我会拿什么数据说服他？"* 如果答案是“我的直觉”或“感觉需要”——那就说明你还没有完成验证。
