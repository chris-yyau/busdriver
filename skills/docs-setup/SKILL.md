# Docs Setup

Standardize documentation across repos using the proven seatbelt/busdriver template. Ensures every repo has consistent README, CONTRIBUTING, SECURITY, LICENSE, and CHANGELOG files with the same structure, badges, and sections.

## When to Use

- Setting up documentation for a new repo
- Auditing existing docs for consistency across repos
- Fixing doc drift (missing sections, wrong badges, inconsistent structure)
- User says "set up docs", "audit docs", "fix docs", "standardize docs"
- After `ci-pipeline-setup` completes (CI config done, now do doc content)

## Modes

### Audit (default)

Check existing docs against the standard. Report what's missing, inconsistent, or outdated. No files modified.

### Apply

Generate missing docs from templates. Fix existing docs to match the standard. Preserves project-specific content inside `<!-- docs-setup:custom -->` markers.

## Process

### 1. Detect Project Context

```bash
# Auto-detect from repo
PROJECT_NAME=$(basename "$(git rev-parse --show-toplevel)")
GITHUB_REMOTE=$(git remote get-url origin 2>/dev/null || echo "")
GITHUB_OWNER=$(echo "$GITHUB_REMOTE" | sed -E 's|.*github\.com[:/]([^/]+)/.*|\1|')
GITHUB_REPO=$(echo "$GITHUB_REMOTE" | sed -E 's|.*github\.com[:/][^/]+/([^/.]+).*|\1|')

# Detect language from files
ls package.json pyproject.toml go.mod Cargo.toml Package.swift build.gradle.kts pom.xml composer.json 2>/dev/null

# Detect component types (Claude Code plugin)
ls -d skills/ agents/ hooks/ commands/ 2>/dev/null

# Check existing docs
ls README.md CONTRIBUTING.md SECURITY.md LICENSE CHANGELOG.md 2>/dev/null
```

### 2. Audit Checklist

For each file, check existence AND content structure:

| File | Checks |
|------|--------|
| README.md | Badges (CI, License, Scorecard), tagline, "What it does" table, Install, Architecture diagram, Skip/bypass, Contributing link, Security link, License line |
| CONTRIBUTING.md | Dev setup (numbered steps), Project structure (ASCII tree), Commit messages (conventional commits), PR process (5 steps), Adding components guide, Code style, Security link |
| SECURITY.md | Contact method, Response timeline (48h/1w/30d), Scope list, Out of scope, Recognition |
| LICENSE | Exists, correct year, correct holder |
| CHANGELOG.md | Exists (auto-generated by semantic-release) |

**Badge format check:**
```
[![CI](https://github.com/{OWNER}/{REPO}/actions/workflows/tests.yml/badge.svg)](https://github.com/{OWNER}/{REPO}/actions/workflows/tests.yml)
[![Security](https://github.com/{OWNER}/{REPO}/actions/workflows/security.yml/badge.svg)](https://github.com/{OWNER}/{REPO}/actions/workflows/security.yml)
[![License: MIT](https://img.shields.io/badge/License-MIT-blue.svg)](LICENSE)
[![OpenSSF Scorecard](https://api.scorecard.dev/projects/github.com/{OWNER}/{REPO}/badge)](https://scorecard.dev/viewer/?uri=github.com/{OWNER}/{REPO})
```

**Content structure checks (grep-based):**
```bash
# README sections
grep -c '^## What it does' README.md
grep -c '^## Install' README.md
grep -c '^## How it works' README.md
grep -c '^## Contributing' README.md
grep -c '^## Security' README.md
grep -c '^## License' README.md

# CONTRIBUTING sections
grep -c '^## Development Setup' CONTRIBUTING.md
grep -c '^## Project Structure' CONTRIBUTING.md
grep -c '^## Commit Messages' CONTRIBUTING.md
grep -c '^## Pull Request Process' CONTRIBUTING.md
grep -c '^## Code Style' CONTRIBUTING.md

# SECURITY sections
grep -c '^## Reporting a Vulnerability' SECURITY.md
grep -c '^## Scope' SECURITY.md
grep -c '^## Out of Scope' SECURITY.md
grep -c '^## Recognition' SECURITY.md

# Badge URLs match repo
grep -c "github.com/${GITHUB_OWNER}/${GITHUB_REPO}" README.md
```

**Cross-file consistency checks:**
```bash
# .releaserc.json has @semantic-release/git plugin (for CHANGELOG auto-commit)
grep -q '@semantic-release/git' .releaserc.json 2>/dev/null

# SECURITY.md email matches across repos (if user has a standard email)
grep -oP '(?<=Email:\*\* ).*' SECURITY.md

# SECURITY.md SLA is standard (48h/1w/30d)
grep -q '48 hours' SECURITY.md
grep -q '1 week' SECURITY.md
grep -q '30 days' SECURITY.md

# Badge count matches standard (4: CI, Security, License, Scorecard)
grep -c 'badge.svg\|img.shields.io\|scorecard.dev' README.md
```

### 3. Present Audit Results

```
| File | Status | Issues |
|------|--------|--------|
| README.md | ✅/⚠️/❌ | Missing "Security" badge, wrong repo URL in Scorecard |
| CONTRIBUTING.md | ✅/⚠️/❌ | Missing "Adding a New Component" section |
| SECURITY.md | ✅/⚠️/❌ | SLA says "2 weeks" instead of "30 days" |
| LICENSE | ✅/❌ | Wrong year |
| CHANGELOG.md | ✅/❌ | Missing — .releaserc.json lacks @semantic-release/git |
```

After showing results, offer to apply fixes for any non-✅ items.

### 4. Apply Templates

When applying, use the templates below. For existing files, preserve content inside `<!-- docs-setup:custom -->` markers.

**Section protection protocol:**
- Standard sections (badges, Contributing, Security, License) are machine-owned — overwrite freely
- Project-specific sections (feature tables, architecture diagrams, component guides) are preserved
- When updating an existing file:
  1. Read the current file
  2. Identify standard sections by heading
  3. Replace standard sections with template versions (variable-substituted)
  4. Preserve everything else (project-specific content between standard sections)
  5. If a section doesn't exist, append it in the correct position

**User input needed for apply:**
- `TAGLINE` — one-line project description
- `DESCRIPTION` — one-paragraph description (can reference existing README)
- `SECURITY_EMAIL` — vulnerability reporting contact
- `COPYRIGHT_HOLDER` — for LICENSE (default: git config user.name)
- `SCOPE_LIST` — what's in scope for SECURITY.md (auto-detected from directory structure)
- `COMPONENT_TYPE` — what contributors can add (scanners, skills, agents, gates, etc.)

### .releaserc.json (ensure CHANGELOG auto-commit)

If `.releaserc.json` exists but lacks `@semantic-release/git`, add it:

```json
{
  "branches": ["main"],
  "plugins": [
    "@semantic-release/commit-analyzer",
    "@semantic-release/release-notes-generator",
    "@semantic-release/changelog",
    ["@semantic-release/git", {
      "assets": ["CHANGELOG.md"],
      "message": "chore(release): ${nextRelease.version} [skip ci]\n\n${nextRelease.notes}"
    }],
    ["@semantic-release/github", {
      "successComment": false,
      "failTitle": false
    }]
  ]
}
```

This ensures CHANGELOG.md is committed to the repo on each release. Without `@semantic-release/git`, the changelog is generated but never persisted.

## Scope Boundary with ci-pipeline-setup

| Concern | Owner | What it does |
|---------|-------|-------------|
| Workflow files (.github/workflows/) | ci-pipeline-setup | Generates/audits CI workflows |
| Dependabot config | ci-pipeline-setup | Generates .github/dependabot.yml |
| Commitlint config | ci-pipeline-setup | Generates commitlint.config.mjs |
| package.json (minimal) | ci-pipeline-setup | Generates private package.json |
| codecov.yml | ci-pipeline-setup | Generates coverage config |
| SHA pin script | ci-pipeline-setup | Generates .github/scripts/check-pinned-uses.sh |
| README.md content | **docs-setup** | Generates/audits doc content |
| CONTRIBUTING.md content | **docs-setup** | Generates/audits doc content |
| SECURITY.md content | **docs-setup** | Generates/audits doc content |
| LICENSE content | **docs-setup** | Generates/audits license |
| .releaserc.json (git plugin) | **docs-setup** | Ensures CHANGELOG auto-commit |
| File existence checks | Both | ci-pipeline-setup checks existence, docs-setup checks content |

Run `ci-pipeline-setup` first (CI config), then `docs-setup` (doc content). They complement each other.

## Troubleshooting

| Problem | Cause | Fix |
|---------|-------|-----|
| Badges show "no status" | Workflow file doesn't exist yet | Run ci-pipeline-setup first to create workflows |
| CHANGELOG.md not auto-generated | .releaserc.json missing `@semantic-release/git` | docs-setup apply fixes this automatically |
| Wrong repo URL in badges | Git remote changed or docs copied from another repo | docs-setup apply regenerates badges from current remote |
| "Adding a New X" section overwritten | Section not inside `<!-- docs-setup:custom -->` markers | Move project-specific content inside markers before re-running apply |
| SECURITY.md email inconsistent across repos | Each repo set up independently | Use docs-setup apply with same `SECURITY_EMAIL` across all repos |

## Key Decisions

- **Separate from ci-pipeline-setup** — CI config is deterministic (generate exact files); doc content is judgment-heavy (templates + project-specific content). Different concerns, different update cadences
- **2 modes, not 3** — "generate" and "fix" are the same operation from the user's perspective. `audit` reports, `apply` acts. Council consensus (2026-03-27)
- **Templates embedded in SKILL.md** — the LLM needs templates in context when generating. Separate files require extra Read calls and add friction. Accepted tradeoff: larger SKILL.md but zero-step template access
- **Section protection via markers** — `<!-- docs-setup:custom -->` / `<!-- docs-setup:end -->` delineate machine-owned vs human-owned content. Standard sections (badges, Contributing link, Security link, License) are always overwritten. Everything between custom markers is preserved. Prevents destructive standardization (Codex council insight, 2026-03-27)
- **4 badges standard** — CI, Security, License:MIT, OpenSSF Scorecard. Security badge was missing from seatbelt (3 badges) — standardized to 4
- **30-day fix SLA standard** — seatbelt had "2 weeks", busdriver had "30 days". Standardized to 30 days (more realistic for solo dev)
- **CHANGELOG auto-commit** — .releaserc.json must include `@semantic-release/git` plugin to persist CHANGELOG.md. busdriver was missing this. docs-setup fixes it
- **Skip CODE_OF_CONDUCT.md and SUPPORT.md** — solo dev with <10 repos. Overhead without community contributions. Reassess at 50+ repos or first external contributor (Codex suggested these; deferred by council verdict)
